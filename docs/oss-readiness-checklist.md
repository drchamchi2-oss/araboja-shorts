# OSS Readiness Checklist

Use this checklist before making the repository public or submitting it to the Codex OSS program.

## Must Do Before Public Release

- [ ] Confirm the repository contains no secrets, API keys, tokens, passwords, cookies, private logs, private datasets, or credentials.
- [ ] Confirm the repository contains no proprietary code, paid assets, private customer data, or third-party files without redistribution rights.
- [ ] Replace placeholder README content with accurate project details.
- [ ] Choose and commit a real open-source license.
- [ ] Add minimal installation and usage instructions.
- [ ] Add a simple working example or command.
- [ ] Add contribution guidelines.
- [ ] Add a security reporting policy.
- [ ] Add issue and pull request templates.
- [ ] Add a changelog or release notes.

## Strong Signals For The Application

- [ ] Public GitHub profile.
- [ ] Public repository.
- [ ] Applicant has admin, maintainer, or clear core contributor role.
- [ ] Recent commits show active maintenance.
- [ ] Issues or PRs show real project operations.
- [ ] The project has users, stars, downloads, dependents, forks, community activity, or clear ecosystem importance.
- [ ] The application text explains the maintainer workload Codex would reduce.
- [ ] API credit plan is limited to eligible OSS maintainer workflows.

## Do Not Claim Unless True

- [ ] Do not claim stars or downloads that do not exist.
- [ ] Do not claim production users without evidence.
- [ ] Do not claim broad ecosystem importance unless the project is actually used that way.
- [ ] Do not claim core maintainer status for a project where you only made minor contributions.
- [ ] Do not request Codex Security for repositories you do not own or administer.

## Suggested Public Release Sequence

1. Run a secret scan locally.
2. Review file history for private content.
3. Finalize README.
4. Finalize license.
5. Add contribution and security policies.
6. Make the repo public only after the above is clean.
7. Wait for visible maintenance activity if the project is brand new.
8. Submit the application with accurate claims.
